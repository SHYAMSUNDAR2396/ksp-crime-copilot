§00 // Executive Summary

One officer, one question, one trustworthy answer

Today an SCRB analyst who wants "all two-wheeler thefts in Bengaluru East linked to a repeat offender in the last six months" must know the schema, write SQL or wait on a dashboard team, and still cannot see the hidden links between FIRs, phones and vehicles. We replace that with a conversation — in Kannada or English — that returns an explainable, source-cited answer in seconds.

This report specifies a Catalyst-native, agentic Retrieval-Augmented Generation (RAG) system over the SCRB crime corpus. It is deliberately two-layered: we present the state-of-the-art landscape (so the jury sees we know the field), and then the exact Zoho Catalyst services each capability maps to — because deployment on Catalyst is mandatory and third-party substitution where a Catalyst service exists can invalidate the submission.

The thesis in one line

A planner-orchestrated multi-agent RAG that fuses structured crime data (Catalyst Data Store), unstructured case documents (QuickML RAG / Knowledge Base) and a relationship graph, served through GLM-4.7-Flash (QuickML LLM Serving) with Kannada-first multilingual handling and citation-level explainability on every answer.

What wins this challenge

  1. Explainability over magic. Every answer shows the FIRs, columns and graph edges it used. Police evidence standards demand traceability; QuickML RAG already exposes a "response breakdown." We make that the centrepiece, not an afterthought.
  2. Hidden-link discovery. A knowledge graph that surfaces the connection between a phone number, three FIRs and a vehicle is the "wow" no dashboard can match.
  3. Kannada-first. A beat officer querying in spoken Kannada and getting a cited Kannada answer is the single most demo-able proof of real-world fit for Karnataka.
  4. Deployability. Running live on Catalyst at demo day — not a localhost mock — is the credibility multiplier with government stakeholders.

§01 // Problem Understanding

Why querying crime data is still painfully hard

The SCRB runs a large, continuously growing repository drawn from 1,100+ stations. As the challenge brief itself states, current systems lean on static dashboards and manual queries, which caps deep analysis and real-time insight. That single sentence hides several distinct bottlenecks:

1.1 The querying bottleneck

  • Schema literacy is a gatekeeper. SQL and BI dashboards require knowing table names, join keys and code lists. A station-house officer should not need to know that "MO" lives in a separate modus-operandi table.
  • Dashboards answer yesterday's questions. A fixed dashboard answers the questions its designer anticipated. Investigations generate novel questions hourly ("who else called this number the night of the burglary?").
  • Data is siloed by record type. FIRs, charge sheets, criminal histories, vehicle records and call data live in different systems and formats; cross-record questions require manual stitching.
  • Unstructured text is invisible to SQL. The richest evidence — the FIR narrative, witness statements, charge-sheet reasoning — is free text that relational queries cannot search semantically.

1.2 The multilingual policing problem (India-specific)

FIRs in Karnataka are frequently registered in Kannada; supervisory and inter-state communication often happens in English or Hindi; border districts see Tamil, Telugu and Marathi. A national crime system that only "speaks" English excludes the beat constable who is closest to the ground truth. Mixed-script and transliterated entries (a name written both in Kannada and Roman script) further fragment search.

1.3 The historical-record problem

Decades of FIRs, charge sheets and case records exist as scanned PDFs and legacy formats. They are effectively unsearchable: an investigator cannot ask "show similar MO cases from the last ten years" because the corpus was never indexed for semantic retrieval.

§02 // Stakeholder & Persona Analysis

Eight users, eight very different questions

The system must serve a constable with a phone in a lane and a Senior Superintendent reading a district trend report. Same engine, different surface, different access rights.

Field · Tier 1

Beat Constable

  • Workflow: patrol, verification, first response.
  • Needs: instant person/vehicle lookup, BOLO alerts.
  • Pain: no field access to records; radios the station.
  • AI opportunity: voice query in Kannada on mobile → cited summary.
Station · Tier 1

Station House Officer (SHO)

  • Workflow: FIR registration, station caseload, daily diary.
  • Needs: jurisdiction caseload, pending actions, similar FIRs.
  • Pain: manual register reconciliation across records.
  • AI opportunity: "summarise today's FIRs and flag repeats."
Investigation · Tier 2

Inspector / IO

  • Workflow: active investigation, evidence assembly.
  • Needs: cross-FIR links, suspect history, MO matches.
  • Pain: stitching phones, vehicles, addresses by hand.
  • AI opportunity: graph link-discovery + auto case brief.
Specialist · Tier 2

Crime Branch Officer

  • Workflow: serious / organised / serial crime.
  • Needs: network maps, repeat-offender clusters.
  • Pain: no tool to reveal hidden criminal networks.
  • AI opportunity: community detection + link prediction.
Analysis · Tier 3

Intelligence Analyst

  • Workflow: trend, hotspot and threat analysis.
  • Needs: spatio-temporal clustering, ad-hoc cohorts.
  • Pain: waits on BI team for every new cut.
  • AI opportunity: self-serve analytics in natural language.
Command · Tier 4

Senior Superintendent (SP)

  • Workflow: district command, resourcing, accountability.
  • Needs: district KPIs, clearance rates, emerging risks.
  • Pain: static reports, lagging signals.
  • AI opportunity: "brief me on this week's district risks."
Cyber · Tier 2

Cyber Crime Officer

  • Workflow: financial fraud, online offences.
  • Needs: link a phone/UPI/account across complaints.
  • Pain: fraud rings span many FIRs and districts.
  • AI opportunity: "all fraud linked to this number" → graph.
Response · Tier 1

Emergency Response (112)

  • Workflow: live dispatch, situational awareness.
  • Needs: address history, prior incidents, nearby risk.
  • Pain: no instant context at point of dispatch.
  • AI opportunity: "what's the history at this address?"
Design consequence

Personas differ on two axes the architecture must respect: access rights (a constable must never see what a Crime Branch officer sees) and answer shape (a one-line field fact vs. a district trend brief). Both are handled by the Orchestrator + RBAC layer, not by separate apps.

4.2 The unified knowledge model

Three layers sit over these sources, all addressable by the agents:

  • Structured layer — normalised entities and facts in Data Store, queried by the Query Agent for precise, aggregatable answers ("how many", "in this period", "in this division").

  • Relationship layer — a graph of entities and edges (modelled as adjacency tables in Data Store, traversed by graph analytics in Functions/QuickML) for "how are these connected" questions.

Honest constraint — and how we handle it

Catalyst does not expose a native graph database. We therefore model the relationship layer as adjacency/edge tables in Data Store and run traversal, community detection and link-prediction as QuickML pipelines and Catalyst Functions. §05 // Conversational AI Design

How an officer actually talks to the system

The interface is a conversation, not a query box. It must understand intent, plan retrieval, hold context across turns, and answer in a way an officer can act on and a court can trust.

5.1 The query types we support

Show all burglary cases in Bengaluru East in the last 6 months Find repeat offenders associated with vehicle theft Summarise all cyber-fraud complaints linked to phone +91-9XXXXXXXXX Show crime hotspots near MG Road in the last 90 days What similar cases exist for this FIR?

These map to four intent families, each routed differently: aggregate/filter (→ Query Agent), semantic/similar (→ Retrieval Agent), relational/link (→ Investigation Agent), and analytic/trend (→ Crime-Pattern Agent).

5.3 Context, multi-turn and explainability

  • Multi-turn context. QuickML's RAG does not retain chat history on its own, so we hold conversation state in Catalyst Cache keyed to the session, letting "now narrow that to two-wheelers" resolve against the previous turn.
  • Context retention scope. The active entities (the FIR, the suspect, the time window) are carried forward; the orchestrator decides what is still relevant.
  • Explainable responses. Every answer renders three things: the plain-language answer, the sources (FIR IDs, table rows, graph edges) via QuickML's response-breakdown, and a confidence/coverage note when data is partial.
Demo gold

Show a follow-up turn live: ask the broad question, then say "only the ones near Indiranagar." The system narrowing correctly using remembered context is the moment judges believe it is a real assistant, not a search box.

§07 // RAG System Research

A law-enforcement-grade retrieval system

Generic RAG returns plausible text. Police RAG must return verifiable text — every sentence traceable to an FIR, a row or a document — and must combine semantic search with exact structured filters.

7.1 The hybrid retrieval principle

Crime questions are rarely purely semantic. "Burglaries in Bengaluru East in the last six months" needs an exact filter (place, type, date) before any semantic ranking. So the system runs hybrid retrieval: the Query Agent applies hard filters in Data Store; the Retrieval Agent does semantic search over the matching case text; results are fused and re-ranked before generation. QuickML's managed RAG already performs semantic search → re-rank → grounded generation with a citation breakdown, which we use directly and wrap with our structured pre-filter.

7.2 The landscape (what we'd reach for off-platform)

Recommended stack — Catalyst-native (the one that counts)

Because deployment is mandated on Catalyst and a managed RAG service exists, the production retrieval stack is QuickML Knowledge Base + RAG (semantic search, re-rank, GLM-4.7-Flash generation, citation breakdown) for unstructured text, + Catalyst Data Store full-text and relational queries for structured pre-filtering. The vendor engines above are presented to demonstrate landscape literacy and as the contingency stack if a self-managed index were ever permitted — not as the submission's runtime.

Two real limitations to engineer around

(1) QuickML RAG currently does not persist chat history — we add multi-turn memory in the app layer via Cache. (2) Retrieval quality depends on chunking and entity-tagging at ingestion — we invest there (FIR-aware chunking, entity tags) rather than expecting the index to do it for us.

§08 // Knowledge Graph Design

Turning records into a network of relationships

The questions only a graph can answer — "who connects these three FIRs?", "what's the hidden link between this phone and that vehicle?" — are the investigative heart of the system.

8.1 Entities and relationships

Crime knowledge-graph schema (illustrative)
flowchart LR
  PER(("Person")) -->|suspected_in| FIR(("FIR / Case"))
  VIC(("Victim")) -->|appears_in| FIR
  WIT(("Witness")) -->|appears_in| FIR
  PER -->|owns| VEH(("Vehicle"))
  PER -->|uses| PH(("Phone"))
  PH -->|called| PH2(("Phone"))
  PER -->|located_at| ADDR(("Address"))
  FIR -->|occurred_at| LOC(("Location"))
  PER -->|holds| ACC(("Bank Account"))
  PER -->|member_of| ORG(("Organisation"))
  FIR -->|related_to| FIR2(("FIR / Case"))

Entities: Person, Criminal, Victim, Witness, FIR, Case, Vehicle, Phone Number, Address, Location, Bank Account, Organisation. Relationships: knows, owns, appears_in, called, located_at, related_to, suspected_in.

8.2 How graph analytics uncovers networks

  • Entity resolution first: collapse "Suresh K", "S. Kumar" and the Kannada spelling into one node — the single highest-leverage step for accuracy.
  • Path-finding reveals indirect links (suspect → phone → co-called number → second FIR).
  • Community detection exposes organised groups and fraud rings.
  • Centrality ranks who is structurally important in a network (the broker, not just the loudest name).
  • Link prediction suggests probable-but-unrecorded connections for investigators to verify (never to act on automatically).
Catalyst implementation

Nodes and edges live as Data Store tables; traversal and the analytics above run as Catalyst Functions (path/community/centrality) and QuickML (link-prediction as a classification pipeline). For demo scale this is fast; for production scale we document a path to an external graph engine if the mandate ever relaxes.

9.1 Recommended multilingual architecture

A translate–reason–translate bridge that stays on-platform:

  1. Detect & normalise the input language and script (Zia handles 10 Indian languages incl. for OCR of scanned Kannada FIRs).
  2. Pivot to English for the heavy reasoning step (where GLM-4.7-Flash is strongest and entity matching is most reliable), keeping a copy of the original for fidelity.
  3. Reason & retrieve over the corpus.
  4. Generate the answer, then render it back in the user's language, preserving names/IDs verbatim so nothing is mistranslated.
The honest Kannada gap — and the winning move

GLM-4.7-Flash's native Kannada generation is the weakest link. The defensible solution within the mandate is the English-pivot bridge above; if the organisers permit a self-hosted helper, slotting in an AI4Bharat/Sarvam translation model on AppSail lifts Kannada quality sharply. Naming this trade-off openly (rather than pretending the model is perfect in Kannada) is itself a credibility win with a Karnataka jury.

On Catalyst, these run as QuickML pipelines (clustering, classification, anomaly detection, time-series are all native operations) with SHAP-based model explanation already provided by QuickML — which we surface to officers so a risk flag always comes with "why".

Predictive policing — the line we draw

International experience is unambiguous: predictive policing that targets people invites legitimate civil-liberties objections. We restrict prediction to resource and pattern support (where/when to patrol, which cases may be linked) with human officers always in the loop, and we log every analytic output for audit. This is both ethically right and strategically smart in front of a government jury.
